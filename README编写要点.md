# README 编写要点

本文档说明 **misc-api 根目录 `README.md`** 应如何编写与维护。  
结构参考自成熟项目的 README 范本（本地文件 `README - 副本.md`，仅作对照，**不提交仓库**）。

---

## 一、文档定位

| 文档 | 读者 | 用途 |
|------|------|------|
| **README.md** | 使用者、部署者、二次开发者 | 了解项目、安装部署、功能入口、版本变更 |
| **README编写要点.md** | 维护者 | 规范 README 结构与写法 |
| **发布检查清单.md** | 维护者 | 发版前打勾 |
| **Gitee推送与发行流程.md** | 维护者 | 推送、打包、发行操作 |
| **发行说明/misc-apiX.Y.Z.md** | 所有人 | 单版本详细变更 |

**原则：** README 面向**对外展示**，写「项目是什么、能做什么、怎么用」；发版命令、Token、内部流程**不要**堆在 README 正文，链到维护文档即可。

---

## 二、推荐章节顺序（固定模板）

按以下顺序组织，章节之间用 `---` 分隔：

```
1. 标题 + 居中标语 + Badge 徽章
2. 项目简介（一段话 + 主要能力列表）
   └─ 可选：UI 规范 / 特色子节（有显著交互规范时）
3. 代码仓库与下载（表格）
4. 功能列表（路径对照表）
5. 后台框架特性（架构与体验要点）
6. 环境要求
7. 目录结构（树形 + 关键文件注释）
8. 安装说明（ numbered list）
9. 伪静态 / URL 重写（Apache + Nginx）
10. 在线更新（用户视角，只说「云端」）
11. 版本记录（按版本倒序，每版一个小节）
12. 开源协议（摘要 + 链接 LICENSE）
13. 作者与仓库
14. 维护文档链接（一行即可）
```

新增大模块（如 API 接口、新业务目录）时，在 **功能列表**、**目录结构**、**版本记录** 三处同步补充。

## 三、各章节写法要点

### 3.1 标题区

```markdown
# misc-api

<p align="center">
  <strong>一句标语，概括 3～4 个核心能力</strong>
</p>

<p align="center">
  <img src="https://img.shields.io/badge/version-1.0.0-blue" alt="version">
  ...
</p>
```

- **项目名**与仓库名一致（`misc-api`）
- **version 徽章**必须与 `core/version.php` 中 `VS_VERSION` 一致
- 发版时**第一件事**改徽章版本号

### 3.2 项目简介

- 第一段：**一句话**说明项目类型 + 部署方式 + 核心价值
- **主要能力：** 用 `-` 列表，每条一个能力点，动词开头，具体可感知
- 避免：「由某某框架迁移」「从某某平台拉取」等内部背景
- 更新机制统一写 **「云端在线更新」**，不写具体 Git 平台名称

### 3.3 代码仓库与下载

| 必填列 | 说明 |
|--------|------|
| 平台 | Gitee 主仓库、发行版下载 |
| 链接 | 可点击的 Markdown 链接 |
| ZIP 命名规则 | `misc-api{版本号}.zip` |
| 详细说明位置 | 指向 `发行说明/` |

### 3.4 功能列表（重要）

用**表格**，三列固定：

| 功能 | 路径 | 说明 |

- **功能**：中文名称，与后台菜单或页面标题一致
- **路径**：以 `/` 开头的 Web 路径
- **说明**：一句话，写用途而非实现细节

每新增一个 `admin/*.php` 或前台页面，**必须**在此表增加一行。

### 3.5 目录结构

- 使用代码块树形结构，根目录名为 `misc-api/`
- 关键文件右侧 `# 注释` 说明职责
- 不要省略 `LICENSE`、`update.json`、`发行说明/` 等根级文件
- 子目录只展开到**使用者需要知道的层级**（不必列出每个 vendor 文件）

### 3.6 安装说明

- 4～5 步 numbered list，从上传到登录
- 写清：目录权限、空数据库、访问 `/install/`
- 不要混入发版、Git 操作

### 3.7 在线更新（对外表述）

**必须遵守：**

- 只说「云端」「远程版本清单」
- 不写 Gitee / GitHub API、Token、仓库 raw 地址
- 写清保护规则：不覆盖 `database.php`、`data/` 等
- 写清分步流程与失败时的手动下载方式（链到发行页即可）

**维护者须知（可简要提及，不展开命令）：**

发版时需同步 `core/version.php`、`update.json`、`update-log.json`；细节见 `发布检查清单.md`。

### 3.8 版本记录

每个版本一个小节：

```markdown
### v1.0.1（2026-07-15）

**类型：** 小版本修复 / 大版本功能

**变更说明：**

- 变更点 1
- 变更点 2
```

- **倒序排列**（最新在上）
- 内容与 `update-log.json`、`发行说明/misc-apiX.Y.Z.md` **保持一致**
- 发版时**必须**追加；仅改代码不记版本 = README 不合格

### 3.9 开源协议

- 链接根目录 `LICENSE`
- 保留「您可以 / 作者声明（免责）」两块摘要
- 完整条文不复制进 README，只链到 LICENSE

### 3.10 作者与仓库

- 仓库主页链接
- Issues 反馈方式

## 四、发版时 README 检查项（必做）

发版前对照 [发布检查清单.md](发布检查清单.md)，README 专项：

- [ ] 标题区 **version 徽章** = `core/version.php`
- [ ] **项目简介 / 主要能力** 反映本版真实功能（无过时描述）
- [ ] **功能列表** 路径与说明与代码一致
- [ ] **目录结构** 含新增目录/文件
- [ ] **版本记录** 已追加本版小节（类型 + 变更说明）
- [ ] 无「迁移」「内部框架名」等不应公开的文字
- [ ] 在线更新章节仍使用「云端」表述
- [ ] 与 `发行说明/misc-api{版本}.md` 变更点一致

## 五、禁止与建议

### 禁止

| 禁止 | 原因 |
|------|------|
| 在 README 写 Gitee Token、发版命令 | 属维护文档，且 Token 敏感 |
| 写「从 Gitee/GitHub 拉取更新」 | 对外统一「云端」 |
| 写项目迁移、fork 来源 | 与当前产品无关 |
| 功能列表与真实路径不一致 | 误导部署与二次开发 |
| 发版不更新版本记录 | 用户无法知晓变更 |

### 建议

| 建议 | 说明 |
|------|------|
| 大功能改版 → 大版本号 + 扩充「主要能力」 | 见版本递增规则 |
| 小修复 → 小版本号 + 只改版本记录 | 简介可不动 |
| 定期通读全文 | 删除已实现「占位」或已下线功能的描述 |
| 同步 Gitee 仓库简介 | 取 README 首段 + 主要能力压缩为一两句话 |

## 六、参考对照

本地 `README - 副本.md`（VanShine 项目）可作为**结构范本**，对照关系：

| 副本章节 | misc-api 对应 |
|----------|---------------|
| 项目简介 + 主要能力 | 同结构，内容按 misc-api 裁剪 |
| 代码仓库与下载 | 仅 Gitee 主仓库（无镜像时可省略 GitHub） |
| 功能列表 | 按当前 admin 页面填写，无文件/CDN 模块则不加 |
| 后台框架特性 | 保留通用描述 |
| 目录结构 | 按 misc-api 实际目录树 |
| 在线更新 | 只说云端，ZIP 名 `misc-api{版本}.zip` |
| 版本记录 | 从 v1.0.0 起逐版追加 |
| 开源协议 | 链到 misc-api 自定义 LICENSE |

**注意：** `README - 副本.md` 为本地参考，已加入 `.gitignore`，请勿提交到 misc-api 仓库。

## 七、维护节奏

| 时机 | README 动作 |
|------|-------------|
| 新增页面/模块 | 更新功能列表、目录结构、简介能力点 |
| 每次发版 | 改徽章版本、追加版本记录、核对发行说明 |
| 大版本重构 | 重写简介与主要能力，检查伪静态/安装是否变化 |
| 季度或重大变更后 | 通读全文，优化 Gitee 仓库描述 |
